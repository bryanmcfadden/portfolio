Modals, Dialogs & Utilities

Modal windows are often called heavy windows or modal dialogs because the window is often used to display a dialog box. Modal windows are commonly used to obtain user confirmation, input, display and emergency states. They are often used when a user is forced to make an important decision.

The dialog window itself is an overlay that is directly positioned over the users viewport. This prevents the user from interacting with elements underneath until interaction has occurred. The most common types of modal dialogs are the alert, input, confirmation and custom dialogs which are explained in the sections below.

Alerts

Overview

Used primarily to warn or tell the user something after an action has occurred.
Sometimes they can be used to inform the user whether the system was successful in completing a task or when the system runs into an error which prevents it from completing a task. Alerts cannot be used to generate any dynamic output or customize output since they cannot accept user input.
They provide a single "OK" button for interaction with the user.

Appropriate Use

Some uses for an alert would be:

  • Incorrect information entered in a form (client side validation)
  • An invalid result from a calculation
  • A warning that a service is not available at a certain date

Design Breakdown

Alerts are predefined within TDC allowing allowing only a single variable:

  • Content

An overlay will cover the entire users screen, while placing the alert dialog within the center of the overlay.
Alerts are auto-generated and cannot be positioned.

Overview

Confirmations allow the user to confirm an action which has been initiated by the user. Examples of using this can be but are not limited to confirming deletion of an item or confirming that the information filled is correct. This is used when a confirmation is needed to prevent an error or to confirm that this is the course of action the user wants to take.

Appropriate Use

Some uses for an Confirmation would be:


  • User validation of a file deletion
  • Continued advancement into next step within a multi-step process
  • Validation of a custom user action within a web application

Design Breakdown

Confirmations are predefined within TDC allowing allowing these variables:

  • Dialog Title
  • Content
  • Button definitions (OK, Cancel, Continue)

An overlay will cover the entire users screen, while placing the confirmation dialog within the center of the overlay.

No notes are available for the confirmation dialog at this time.

Tooltips

Click For Tooltip
  • About
  • Guidelines
  • HTML & CSS
  • Notes

Overview

Tool tips provide additional contextual information about a page elements or control and are activated on tap. The help tooltip appears when information could be confusing to some users and requires clarification. When pressed, an overlay appears with the additional information. To dismiss the overlay users can either press anywhere outside the overlay, or press the help button again. Tooltips of this type may be positioned to open above or below. The default position is to open below.

Appropriate Use

The tooltip control may be used inline with controls such as toggle, slider, radio and checkbox. This control should not be used with input fields (see Text input field helpers).

Design Breakdown

Placement

The placement of a tooltip can vary based on the location of the icon that it’s triggered from. It will appear either above or below the icon, pushing down other content that may exist on the page.

Size

The height of a field helper varies based on the content that it contains, but the width is fixed to the viewport, with no padding on either side.

Content

Tooltip text should be as concise as possible while providing benefit to the user. Character count: 150-175 characters max (with spaces)

Interactions

When open, a tool tip pushes other content down the page. A user can tap anywhere on the screen to dismiss the tool tip.
