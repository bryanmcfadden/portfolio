Forms, Input & Selections

Test Test Test Test Test Test Test Test Test Test Test Test

Autocomplete

Implement an autocomplete field to facilitate a filter or search functionality when a significant amount of data is being handled. As the user enters text in this field, relevant results populate in the suggestions box directly beneath it. This component allows users to perform a quick search, so they can find what they need effortlessly.

Checkboxes

Description

This type of component is used when you want to allow the user to choose anywhere from none to many or all options. If you want the user to choose only one option, see Radio Buttons. We strongly recommend that this component is set so that it is unchecked initially. So when the user first clicks on a checkbox it does get checked.


Placement

Arrange checkboxes in a column, so that they are stacked vertically. If you have a 5arge number of options it is acceptable to have two or three columns of checkboxes.

Content

A checkbox should always have a clear and concise label next to it. The first word of the label should be capitalized. No period is needed at the end of a label.

You can also add directions before the grouping of checkboxes instructing the user to select all that apply; please note that this is optional.

Interactions

When a user clicks on a checkbox its state should change to checked or unchecked depending on its previous condition. Clicking on a checkbox can display or hide another component. For example, you have a checkbox that states “Started new project” and by checking it, a text field gets displayed beneath it where the user is required to enter the project name.

Date Field

Description

The date field is commonly used to gather event or activity dates from the user. Use this component when you want users to enter information about when they performed or will perform a certain action or task. You may also use two fields to get beginning and end dates for a task.

Placement

Left-align this field. If you have two fields, you may arrange them in two columns.

Content

A short and understandable label describing the date field must be listed above it. For example, if you want the user to input a date for which he or she began a project you may have a label that says “Project Start Date:” Note that as with most labels for input fields, this label must end with a colon.

Interactions

Users should be able to enter dates in mm/dd/yyyy format or select a date from the calendar pop-up.

Form Layout

Description

If you want to obtain several pieces of information from a user via input then you will need a form. Forms can contain various elements to collect input.

When a form is lengthy, you should provide a way for the user to save his or her data; additionally, if you have split your form into several sections through which users need to navigate, implementing a “Continue” button is appropriate.

Placement

All input elements must be left-aligned. If there are too many elements and/or you want to group them in a way where a one-column layout does not make sense, it is acceptable to arrange them in two columns.

Content

Forms can contain any kind of input element including: text fields, text areas, date fields, checkboxes, radio buttons, document upload functionalities, and drop down menus. Fields are required to have labels informing the user about the input expected to be entered.

Buttons supporting the functionalities to cancel, save, submit, continue, and attach documents can also be included in a form.

Interactions

A form should always have functionalities to both submit and cancel it. If you want users to upload documents, the form must have a way for people to attach them.

When a form is lengthy, you should provide a way for the user to save his or her data; additionally, if you have split your form into several sections through which users need to navigate, implementing a link that jumps to the section that follows is appropriate.

Grouping

Description

Grouping refers to the proximity and division of elements. Close proximity between elements implies that they are related to each other while far proximity differentiates elements. Divisions, such as horizontal rules or rectangles that encompass certain elements, can be used both to create a grouping of elements and set them apart from other sections.

Placement

TDC mainly uses left-aligned section headings with horizontal rules to group elements.

Content

We use section headings and horizontal rules to create different groupings. Please see Headings for more information. Horizontal rules go all the way across the form’s container and lie 10px below their corresponding section heading. See Horizontal and Vertical Rules for more information. Each section may contain any number of related input elements.

Labels

Description

All fields within a form are required to have a relevant descriptor about the type of input the user is expected to enter. Labels must always be clear and concise. They should also be in title case (i.e., using “Project Name:” instead of “Project name:”).

Placement

A label should be aligned to its corresponding input field should be positioned above it.

Radio Buttons

Description

Use this component if you want the user to choose only one option. If you want to allow the user to choose multiple or no options see Checkboxes. We strongly recommend that this component is initially disabled for all of the options.


Placement

Arrange radio buttons in a column, so that they are stacked vertically. If you have a large number of options it is acceptable to have two or three columns of radio buttons.

Content

A radio button should always have a clear and concise label next to it. It must be positioned to the right of the radio button. The first word of the label should be capitalized. No period is needed at the end of a label. Labels must be #666666, 12px, and ClearviewATT Light.

You must always include directions before a grouping of radio buttons instructing the user what they need to select.

Interactions

When a user clicks on a radio button its state should change to enabled or disabled if its previous state was disabled or enabled, respectively. Clicking on a radio button can display or hide another component. For example, you have a radio button that states “Completed code review” and by enabling it, a date field appears beneath it where the user is required to enter the date when the code review was completed.

Text Area

Description

A text area is similar to a text field, but can handle multiple lines text. This input control is implemented when the user is expected to type more than one line of text or over 125 characters.

Placement

Left-align this field.

Content

This element should have a label above it. See Labels for more details.

Interactions

When the user clicks on this component, he or she must be able to type text in it. Users should be allowed to type more than 50 characters in this component.

A text area may also have a handle in the lower right corner that users may click and drag to resize it, so they can more easily view the text they have typed.

Text Field

Description

In contrast to the text area, the text field is one-line input field. Use this when the user is expected to type up to 125 characters.


Placement

Left-align this field.

Content

This element should have a label above it. See Labels for more details.

Interactions

When the user clicks on this component, he or she must be able to type text in it. No more than 50 characters should be allowed to be typed into this kind of input element.

Text Field Helper

Description

Hints or placeholder text are used within an input field to give the user a better idea of what they should enter within the field. For example, if you have a text field where you expect the user to enter a PRISM ID you may use a help text that appears within the field which says “ex. 123456.”


Placement

Placeholder text should be placed within the input element.

Content

This is an alternative to using tooltips to supply additional information. Use this kind of feature only if you want to provide examples of what the user may enter (see the example provided under Description). Otherwise, giving instructions through this type of component may sometimes hurts usability when the user goes back to review their filled in information but can no longer see the hint.

Interactions

As long as a user has not entered text within a field, placeholder text is always listed within an input element. When the user clicks on this component, the placeholder text disappears and the user should be able to start typing within it.

Validation

Description

Validation is used to ensure that the data entered by the user is as correct as possible. For example, if you want a user to enter a project number, you will want to check that the user is only entering digits in the field. If the user includes letters or special characters in his or her entry then an error appears.

You can validate data and display an error message in these scenarios:

  • In real-time, as the user is typing and inserts an invalid character.

  • The user has filled-in a field and navigated to another one and they have made a mistake in the completed field.

  • The user submits his or her form and there are any invalid data types within any of the fields.
